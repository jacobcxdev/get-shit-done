# Phase 6 Claude First Pass

## Recommended Gray Areas

### `check-slim-eligibility` as a compile subcommand flag vs. a standalone query

- Decision: Implement `--check-slim-eligibility <workflow-id>` as a flag on the existing `gsd-sdk compile` command, not as a new subcommand or a `gsd-sdk query` surface. It should reuse the compiler's manifest pipeline (typed transitions from `workflow-semantics`, packet sequencing from `packet-contracts`, provider routing from `routing.ts`, parity coverage from baselines) and emit a structured eligibility verdict to stdout with a non-zero exit on any failed gate.
- Rationale: The compiler already collects and validates all four eligibility inputs — workflow semantic manifests, packet definition candidates, provider routing metadata, and baseline coverage. Adding a flag that re-uses `runCompiler()` output and applies a per-workflow eligibility filter is a one-pass operation with no new inventory logic. A new subcommand or query surface would duplicate corpus collection and create a second code path that could drift from the compiler's canonical output. The existing `CompileReport` and `ClassificationEntry` types already carry the `isHardOutlier` and `outlierPosture` fields needed for eligibility decisions. The `COMPILE_USAGE` help string already scopes compile as repo-wide; per-workflow filtering is a post-process on that same corpus.
- Planning implications: The plan must add `checkSlimEligibility?: string` to `ParsedCompileArgs` and `CompileOptions`, wire it into `runCompiler()` as a post-validation filter that inspects the named workflow's semantic manifest and baseline coverage, and emit a typed eligibility report. The parity gate (`scripts/phase4-parity.cjs`, step `sdk-compile-check`) already runs `gsd-sdk compile --check`; the new flag should be independently exercisable and tested with a dedicated CJS test under `tests/` that asserts exit codes for eligible and ineligible workflows.

### Slim transition: launcher-only Markdown vs. full archive — sequencing and reversibility

- Decision: The slim transition must be strictly two-step and irreversible within a PR. Step 1 produces a launcher-only Markdown file (SLIM-02 posture: advisory invocation only, no inline prose). Step 2 moves the original to `docs/archive/` (SLIM-01 posture: full supersession confirmed). These two steps must land in separate commits so that a CI failure at step 2 leaves the repo with a valid launcher rather than neither. The launcher file must remain as the canonical source even after archiving the original — it is the human-readable entry point for the command.
- Rationale: The prior decision that "Markdown is compatibility/human-planning surface, not runtime contract" means the launcher cannot be silently deleted; it serves human readers invoking `/gsd-plan-phase` etc. from a skills file or README. Bundling both steps into one commit makes rollback (via `gsd-sdk undo`) require restoring a deleted file from git, which is error-prone. Keeping step 1 and step 2 as separate commits lets the phase manifest track each as an independent plan with its own artifact assertion, matching the existing pattern in phases 1–5 where each plan has distinct `files_modified` entries. SLIM-03 already requires CI to block cleanup PRs until parity exits zero; step separation makes the CI gate checkpoints explicit at commit granularity.
- Planning implications: Each workflow eligible for slimming needs exactly two plans: one for the launcher reduction and one for archiving. Plans should share the eligibility check output as a precondition artifact (written to `.planning/compile/slim-eligibility-<workflow-id>.json`) so the archive plan can assert the eligibility verdict was previously green. The `docs/archive/` directory needs to exist before the first archive plan runs; an initialisation plan or prerequisite check must create it.

### Hard outlier posture record format and storage location

- Decision: Each hard outlier posture record is a YAML-frontmatter file stored under `sdk/src/advisory/outlier-postures/<command-id>.yaml` (e.g. `gsd-graphify.yaml`), with fields: `commandId`, `classifiedAs: hard-outlier`, `migrationDisposition`, `rationale`, `emitsPacket: false`, and `reviewedAt`. The existing `ClassificationEntry.outlierPosture` field in `compile/types.ts` stores the string digest of the rationale for manifest consumers. The compiler reads the YAML files and populates `outlierPosture` during `classifyCommands()`, failing with CLSS-04 if any seed hard outlier lacks a posture file.
- Rationale: The `ClassificationEntry` type already has an `outlierPosture?: string` field, and `SEED_HARD_OUTLIERS` in `classification.ts` already enforces that any new hard outlier outside the seed set triggers CLSS-04. The missing piece is materialising a human-readable posture record for each of the five seeds. Storing posture records as YAML files under `sdk/src/advisory/outlier-postures/` keeps them co-located with the advisory module, gives them a stable path the compiler can discover via glob, and makes them diffable in PRs. Inline comments in `classification.ts` are insufficient for OUTL-01 because they are not machine-readable and cannot be surfaced via `gsd-sdk compile --json`. A separate directory in `docs/` would separate posture records from the type that consumes them, making compiler wiring harder to audit.
- Planning implications: Five YAML posture files must be created — one per seed hard outlier. The compiler's `classifyCommands()` must be extended to glob `sdk/src/advisory/outlier-postures/*.yaml`, parse each file, and populate `ClassificationEntry.outlierPosture` for matching entries. A diagnostic `OUTL-01` (new code, not conflicting with existing COMP-*, BILL-*, PCKT-*, EXT-*, MIGR-*) must be emitted for any seed outlier missing a posture file. The parity baseline for `command-classification` must be regenerated after posture files land.

### CI gate for cleanup PRs: existing phase4-parity.cjs vs. a new SLIM-03 gate

- Decision: SLIM-03 is satisfied by extending `scripts/phase4-parity.cjs` with a new wave-0 step that runs `gsd-sdk compile --check-slim-eligibility <workflow-id>` for each workflow whose launcher file has been reduced but not yet archived. This step should be conditional: it only runs when `.planning/compile/slim-eligibility-*.json` files exist in the repo, so the gate is a no-op until Phase 6 work begins and does not affect Phase 1–5 parity runs.
- Rationale: Phase 4's `phase4-parity.cjs` is already the hermetic gate referenced by Phase 5 (must_haves truth: "Phase 4 parity gate passes after all Phase 5 additions"). Adding a new separate gate script would introduce two gate entry points that could diverge in CI configuration. Extending the existing script with a conditional step preserves the single-gate model while satisfying SLIM-03. The conditional on eligibility JSON file existence means Phase 1–5 guarantees are structurally unchanged — the new step is only reachable when Phase 6 cleanup work is in progress, which is consistent with the prior decision that "no Phase 6 work may weaken Phase 1–5 guarantees."
- Planning implications: The plan must add a new step function to `phase4-parity.cjs` that globs `.planning/compile/slim-eligibility-*.json`, extracts workflow IDs, and runs `--check-slim-eligibility` for each. The step must be ordered before `sdk-compile-check` so a failed eligibility check surfaces before the baseline comparison. A CJS test (`tests/slim-eligibility-gate.test.cjs`) must assert that the gate step is present in the script and that it is conditional on file existence.

## Additional Questions

- **Scope of slimming in Phase 6 v2+:** SLIM-01 and SLIM-02 define the eligibility gate and launcher-only posture, but do not specify which workflows are candidates in this phase. Should Phase 6 target a bounded pilot set (e.g. the five workflows with the most complete semantic manifests as measured by `workflow-semantics` baseline) rather than all 84 workflows, to keep the phase atomic and verifiable? A pilot set of 3–5 workflows would produce a reusable slimming pattern validated by CI before broader application.
- **Posture record lifecycle for future hard outliers:** OUTL-02 requires the classification manifest to register hard outliers so `compile` does not flag them as unknown. The current `SEED_HARD_OUTLIERS` set in `classification.ts` is a closed constant. If a future command is reclassified to `hard-outlier` outside Phase 6, is the posture-file mechanism sufficient to extend the set, or does `classification.ts` also need to be updated? The plan should clarify whether the seed set remains the canonical hard-coded list or whether the YAML posture files become the authoritative source that the compiler reads to build the set dynamically.
- **`docs/archive/` ownership:** The `docs/` directory exists in the repo root. Does archiving workflow Markdown files to `docs/archive/` require an update to any documentation parity test (e.g. `tests/workflow-compat.test.cjs`, `tests/commands-doc-parity.test.cjs`, `tests/workflow-size-budget.test.cjs`) that currently asserts a fixed inventory of files under `get-shit-done/workflows/`? The plan must audit which tests enumerate workflow file counts or paths and update them to account for archived files before any archive commit lands.

## Risks and Assumptions

- **Risk — eligibility gate has no prior implementation:** `gsd-sdk compile --check-slim-eligibility` does not exist yet. The four criteria (typed transitions, packet sequencing, provider routing, parity coverage) map onto existing compiler outputs (workflow-semantics, packet-contracts, routing.ts, baselines), but the per-workflow eligibility filter logic must be written and tested from scratch. If any criterion's compiler output is incomplete for a candidate workflow, the gate will always fail, blocking the archive step. The plan must include a fallback: if eligibility cannot be determined from compiler output alone, the gate should emit a `warning` rather than a hard error, with a `--force-eligible` escape hatch for human override.
- **Risk — workflow tests enumerate file paths:** Several CJS tests reference workflow file paths or counts (`tests/workflow-size-budget.test.cjs`, `tests/commands-doc-parity.test.cjs`, `tests/workflow-guard-registration.test.cjs`). Moving any file to `docs/archive/` without updating these tests will break the Phase 4 parity gate. This is a sequencing risk: archive plans must include a sub-task for updating affected tests, and the parity gate must be green before the archive commit is considered complete.
- **Risk — hard outlier posture rationale content:** OUTL-01 requires each posture record to state "classification and migration rationale." For `/gsd-ultraplan-phase` and `/gsd-from-gsd2`, the rationale may require editorial judgment about whether these commands are permanently hard-outlier or temporarily so pending a future migration. The plan should treat rationale content as a human-authored artifact (written by the planner, not generated from manifests) and should not block on compiler automation to produce it.
- **Assumption — Phase 4 parity gate is already green:** STATE.md confirms Phase 5 is complete and all 39 plans executed. The Phase 4 parity gate is assumed to be passing at Phase 6 start. If it is not, Phase 6 plan execution must be blocked until it passes, as SLIM-03 extends this gate.
- **Assumption — `docs/archive/` does not exist yet:** No `docs/archive/` directory was observed in the HEAD tree listing. The first Phase 6 plan must create this directory with a `.gitkeep` or equivalent before any archive plan runs.
- **Assumption — the five seed hard outliers have existing workflow files:** The tree listing confirms `get-shit-done/workflows/fast.md`, `get-shit-done/workflows/review.md`, and `get-shit-done/workflows/ultraplan-phase.md` exist. `/gsd-graphify` and `/gsd-from-gsd2` should have corresponding files (`graphify.md`, `from-gsd2.md` or similar) — this was not verified but is assumed given CLSS-01 confirms all 86 commands are classified and the tree shows `tests/graphify.test.cjs` exists.
