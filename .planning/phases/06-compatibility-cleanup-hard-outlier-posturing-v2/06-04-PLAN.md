    ```javascript
    {
      name: 'slim-eligibility',
      diag: 'SLIM-03',
      run: () => {
        const { readdirSync, readFileSync } = require('fs');
        process.stdout.write('\n[SLIM-03] Running step: slim-eligibility\n');

        // Detect thin launchers: top-level workflow .md files whose entire trimmed
        // content is a single ```gsd-advisory ... ``` fenced block.
        const workflowsDir = join(ROOT, 'get-shit-done', 'workflows');
        let files;
        try {
          files = readdirSync(workflowsDir);
        } catch {
          files = [];
        }

        // Collect workflowIds from launcher metadata (not filename alone).
        const launcherWorkflowIds = [];
        for (const file of files) {
          if (!file.endsWith('.md')) continue;
          let content;
          try {
            content = readFileSync(join(workflowsDir, file), 'utf8');
          } catch {
            continue;
          }
          const trimmed = content.trim();
          // Must be exactly one fenced gsd-advisory block — same rule as extractLauncherBlock
          const m = trimmed.match(/^```gsd-advisory\r?\n([\s\S]+?)\r?\n```$/);
          if (!m || trimmed !== m[0]) continue;
          // Extract workflowId from the block's YAML content
          const idMatch = m[1].match(/^workflowId:\s*(\S+)/m);
          if (idMatch) {
            launcherWorkflowIds.push(idMatch[1].trim());
          }
        }

        if (launcherWorkflowIds.length === 0) {
          process.stdout.write('[SLIM-03] PASS: slim-eligibility (no thin launchers detected — no-op)\n');
          return;
        }

        process.stdout.write(`[SLIM-03] Found ${launcherWorkflowIds.length} thin launcher(s): ${launcherWorkflowIds.join(', ')}\n`);
        for (const workflowId of launcherWorkflowIds) {
          run(
            `slim-eligibility:${workflowId}`,
            'SLIM-03',
            node,
            [join(ROOT, 'bin', 'gsd-sdk.js'), 'compile', '--check-slim-eligibility', workflowId],
          );
        }
      },
    },
    ```

    CRITICAL constraints to verify during implementation:
    1. The step reads workflow files using Node.js built-in `require('fs')` — no SDK imports inside the script
    2. The step calls `node bin/gsd-sdk.js compile --check-slim-eligibility <workflowId>` via execFileSync — this is the ONLY allowed direction (script → SDK). The SDK must NEVER call scripts/phase4-parity.cjs.
    3. The step detection is data-driven: workflowId comes from the launcher YAML block content (`workflowId:` field), not from the filename. This prevents a filename change from silently bypassing the gate.
    4. The step is fail-closed: if execFileSync for any workflowId exits non-zero (fail or indeterminate verdict), the `run()` helper calls process.exit(1) immediately.
    5. The step is a no-op when launcherWorkflowIds.length === 0.
    6. No archive files (docs/archive/*.md) detection is needed in this step — the workflowId from the launcher block is sufficient to call --check-slim-eligibility.
    7. Preserve all existing step entries unchanged — do NOT modify wave0-hardening, hook-build, hook-install-tests, sdk-compile-check, staleness-gate, parity-suite, skip-ban, offline-scan, or retirement-scan.
    8. Do NOT add any require() for SDK modules — the script remains CJS and must not import TypeScript SDK source.

    After modifying the file, verify the step array length increased by exactly one and the new step is at the correct index.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| scripts/phase4-parity.cjs → SDK | Script calls SDK via execFileSync — one-directional. SDK must never call back to the script. |
| Launcher file content → workflowId extraction | workflowId is extracted from file content via regex — must not be executable |
| --step <name> argv → step filter | stepFlag is matched against step.name strings — no eval or dynamic require |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-04-01 | Tampering | launcher file with malicious workflowId | mitigate | workflowId is passed as a positional argument to execFileSync — shell injection is not possible; execFileSync does not invoke a shell |
| T-06-04-02 | Elevation of Privilege | SDK calling scripts/phase4-parity.cjs | mitigate | SDK code (slim-eligibility.ts, outlier-postures.ts, compiler.ts) must not import or spawn scripts/phase4-parity.cjs — enforced by RESEARCH.md D-03 and billing boundary; verified by existing offline-scan step |
| T-06-04-03 | Denial of Service | readdirSync failure on missing workflows dir | mitigate | try/catch around readdirSync; empty files array → no-op |
| T-06-04-04 | Tampering | workflowId spoofing via filename-only detection | mitigate | workflowId is extracted from the launcher YAML content (`workflowId:` field), not from the filename — a renamed file with a different workflowId in content will use the content value |
| T-06-04-05 | Repudiation | slim-eligibility step silently skipped | mitigate | Step always logs either "no thin launchers detected — no-op" (with PASS) or the list of detected launchers before running checks — no silent skip path |
