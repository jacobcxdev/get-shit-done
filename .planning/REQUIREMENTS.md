# Requirements: GSD Advisory FSM Migration

**Defined:** 2026-04-27
**Core Value:** The SDK emits one atomic instruction packet at a time; the runtime executes it and reports back â€” `gsd-post-update` must become unnecessary.
**Roundtable status:** Reduced-confidence requirements roundtable completed 2026-04-27 with Claude + Codex; Gemini Pro unavailable/exhausted and recorded as missing.

## v1 Requirements

### Compiler / Audit Foundation

- [x] **COMP-00**: `gsd-sdk compile` is added as a new SDK CLI subcommand, wired through the same project-root/config resolution path as `gsd-sdk query`, and documented in CLI help
- [x] **COMP-01**: `gsd-sdk compile` produces a command coverage manifest listing all 86 live commands with their IDs, types, workflow associations, and classification
- [x] **COMP-02**: `gsd-sdk compile` produces a workflow coverage manifest listing all 84 workflows with their step counts, runner types, determinism classification, and workflow semantic features
- [x] **COMP-03**: `gsd-sdk compile` produces an agent contract manifest listing all 33 agents with role class, tool permissions, disk-write mandate flags, and worktree requirements
- [x] **COMP-04**: `gsd-sdk compile` produces a hook/install manifest listing every file in `hooks/` and `hooks/dist/` with install target and executability status
- [x] **COMP-05**: `gsd-sdk compile` exits non-zero and names conflicting IDs when duplicate workflow, command, packet, or extension IDs are detected
- [x] **COMP-06**: `gsd-sdk compile` exits non-zero and names the offending `workflowId` and `stepId` when a packet exceeds the complexity/size budget
- [x] **COMP-07**: `gsd-sdk compile` exits non-zero and names the missing workflow when a command references an unknown workflow
- [x] **COMP-08**: `gsd-sdk compile` emits inventory/doc parity warnings for every command, workflow, agent, or hook present in one source but absent from its paired documentation or manifest
- [x] **COMP-09**: `gsd-sdk compile` validates extension dependency ordering, rejects cycles, and reports the dependency chain that caused the cycle
- [x] **COMP-10**: `gsd-sdk compile` validates state references, generated compatibility artefacts, transform ordering, idempotency metadata, and atomic-write declarations before any runtime use
- [x] **COMP-11**: `gsd-sdk compile --assert-counts` exits non-zero when manifest counts differ from the expected seed counts unless the run explicitly records a live-repo count update
- [x] **COMP-12**: `gsd-sdk compile --check-billing-boundary` fails if default advisory entry points import or call Claude Agent SDK `query()` or open SDK-internal model sessions outside an explicit API-key execution mode
- [x] **COMP-13**: all `gsd-sdk compile` manifest outputs are machine-readable JSON suitable for golden comparisons in the parity suite

### Per-Command Classification

- [x] **CLSS-01**: All 86 live commands are classified in a machine-readable manifest under exactly one of: `core-lifecycle`, `composite`, `query-utility`, `single-agent-bounded`, `dynamic-branch`, `hard-outlier`
- [x] **CLSS-02**: The classification manifest is generated by `gsd-sdk compile` from the live repo inventory â€” no entry is hand-authored as a hidden fixture
- [x] **CLSS-03**: Each manifest entry includes command name, classification, associated workflow ID or `null`, invoked agent types, determinism posture, and migration disposition
- [x] **CLSS-04**: `gsd-sdk compile` exits non-zero for any command that matches no taxonomy rule, naming the unclassifiable command

### Billing Boundary

- [x] **BILL-01**: Default advisory commands and runner paths contain no Claude Agent SDK `query()` calls and do not open SDK-internal model sessions for orchestration
- [x] **BILL-02**: `gsd-sdk advise`, `gsd-sdk next`, and `gsd-sdk report` work without Anthropic API credentials and without model/network calls in the default advisory path
- [x] **BILL-03**: Any API-key model execution mode is named explicitly, opt-in, separately authorised, and never selected by default command aliases
- [x] **BILL-04**: A regression test fails if default advisory execution attempts to resolve Anthropic API credentials or instantiate a Claude Agent SDK model session

### Instruction Packet Schema

- [x] **PCKT-01**: Every SDK-emitted instruction packet contains all required fields: `schemaVersion`, `runId`, `workflowId`, `stateId`, `stepId`, `goal`, `instruction`, `requiredContext`, `allowedTools`, `agents`, `expectedEvidence`, `allowedOutcomes`, `reportCommand`, `onSuccess`, `onFailure`, `checkpoint`, `configSnapshotHash`, `extensionIds`, `executionConstraints`
- [x] **PCKT-02**: A packet with a missing required field is rejected at compile time with the field name, `workflowId`, and `stepId` included in the error message
- [x] **PCKT-03**: A packet whose `schemaVersion` does not match the SDK's current schema version is rejected with a version-mismatch error at load time
- [x] **PCKT-04**: A packet whose `instruction` spans multiple logical actions is rejected by the compiler as coarse
- [x] **PCKT-05**: `configSnapshotHash` is the SHA-256 digest of `.planning/config.json` at packet-emission time, verified by a parity test that mutates config and checks the hash changes
- [x] **PCKT-06**: The packet schema is published as a TypeScript type in the SDK with one-line JSDoc for each field
- [x] **PCKT-07**: `executionConstraints` can encode foreground-only execution such as Codex relay `run_in_background: false`
- [x] **PCKT-08**: A packet targeting a disk-write-mandate agent without a disk-write capability in `allowedTools` is rejected at compile time using the Phase 1 agent contract manifest

### Config Validation and Routing

- [x] **CONF-01**: The SDK validates `.planning/config.json` at startup and exits with a descriptive error naming missing or malformed `agent_routing`, `codex_model`, `gemini_model`, `codex_config`, or `gemini_config` fields when those fields are required by the selected profile
- [x] **CONF-02**: `agent_routing` values of the form `codex:<effort>` are parsed and validated at startup; unrecognised effort levels produce startup errors, not runtime failures
- [x] **CONF-03**: Step-scoped `agent_routing` keys of the form `<agent-name>.<workflow-id>.<step-id>` override agent-level defaults for that transition only
- [x] **CONF-04**: Routing resolution is config-snapshot-aware and happens per packet emission, not once at runner initialisation
- [x] **CONF-05**: `gsd-sdk query config.snapshot-hash` returns the SHA-256 hash of the current `.planning/config.json` content
- [x] **CONF-06**: New config fields introduced by the FSM migration are added to both `sdk/src/query/config-schema.ts` and `get-shit-done/bin/lib/config-schema.cjs` in the same change, verified by config schema parity tests
- [x] **CONF-07**: Project config preserves `commit_docs: false`, `constraints.commit_planning_artifacts: false`, and `constraints.planning_artifacts_ignored: true` for this migration project

### Durable State Model

- [x] **STAT-01**: Each workstream's FSM run-state is stored under the existing `planningPaths` convention: flat mode uses `.planning/fsm-state.json`; workstream mode uses `.planning/workstreams/<workstream>/fsm-state.json`
- [x] **STAT-02**: Every FSM run-state file includes `runId`, `stateSchemaVersion`, current state, config snapshot hash, and transition history from its first write
- [x] **STAT-03**: `gsd-sdk query check.auto-mode <workstream>` returns the auto-mode flag scoped to the specified workstream, not a global shared value
- [x] **STAT-04**: `workflow._auto_chain_active` no longer exists as a flat shared key; its semantics are captured per-workstream inside FSM run-state
- [x] **STAT-05**: `STATE.md` mirror sync is executed only while the workstream FSM write-lock is held; a concurrent sync attempt returns a typed `lock-conflict` error
- [x] **STAT-06**: `gsd-sdk query lock.status <workstream>` returns lock holder, acquisition timestamp, age, and `stale: true/false`
- [x] **STAT-07**: FSM run-state files are valid JSON, human-readable, and produce clean text diffs
- [x] **STAT-08**: The human-readable `STATE.md` format is byte-identical to its pre-migration format when the FSM runner writes it
- [x] **STAT-09**: New FSM state read/write paths route through SDK query state modules; CJS fallback via `gsd-tools.cjs` is not canonical for new FSM surfaces
- [x] **STAT-10**: New FSM state writes and FSM-adjacent init writes do not use silent catch blocks; failures rethrow, emit typed events, or log at warn/error level

### Workflow Semantics

- [x] **WFSM-01**: Workflow manifests encode mode/flag priority dispatch for workflows with multiple invocation modes
- [x] **WFSM-02**: HITL suspension states are represented explicitly with resumable packet outcomes and testable mock input seams
- [x] **WFSM-03**: Config-gated subgraphs are represented as deterministic guards with named config dependencies
- [x] **WFSM-04**: Context-budget-aware branching is represented through deterministic context budget inputs, not model judgement
- [x] **WFSM-05**: Parallel wave lifecycle, filesystem fallback, sentinel-file polling, and non-blocking side-effect states are represented in workflow manifests
- [x] **WFSM-06**: Nested FSMs for debugger, node repair, review/revision loops, and verifier escalation are represented with parent/child run-state links
- [x] **WFSM-07**: Dynamic roadmap re-read and workstream namespace injection are represented as explicit transition inputs
- [x] **WFSM-08**: Runtime-type branching, AskUserQuestion/text-mode substitution, quantitative gates, and completion-marker checks are represented without ad hoc prose patching

### Agent Contract Registry

- [x] **AGNT-01**: The Phase 1 agent contract manifest enumerates all 33 agents with role class, allowed tools, `diskWriteMandate`, `worktreeRequired`, output artefacts, and completion marker
- [x] **AGNT-02**: The 14 disk-write-mandate agents are marked in the manifest and used by packet validation before any runner migration
- [x] **AGNT-03**: Missing completion markers after an agent reports success emit typed `completion-marker-missing` events before parity testing
- [x] **AGNT-04**: A packet targeting a `worktreeRequired` agent when no worktree is active emits `worktree-required` before execution is requested

### Workflow-Agnostic Advisory Runner

- [x] **RNNR-01**: The advisory runner supports the classified workflow patterns from the compiler manifest rather than assuming every workflow follows the existing phase chain
- [x] **RNNR-02**: The existing `PhaseRunner` canonical phase chain (`discuss â†’ research â†’ plan â†’ plan-check â†’ execute â†’ verify â†’ advance`) is preserved with identical transitions, locking behaviour, and emitted packet sequence
- [x] **RNNR-03**: The existing `InitRunner` behaviour is preserved with identical deterministic transitions while removing default model-backed orchestration from the advisory path
- [x] **RNNR-04**: Nyquist P4 compliance is encoded as a distinct FSM state (`p4-compliance`) after `verify` and before `advance`, not merged into `verify`
- [x] **RNNR-05**: The runner emits exactly one instruction packet per state transition â€” no batch emissions
- [x] **RNNR-06**: The runner contains no tool-execution calls; it only emits packets and awaits outcome reports from the runtime
- [x] **RNNR-07**: Hard outliers (`/gsd-graphify`, `/gsd-from-gsd2`, `/gsd-ultraplan-phase`, `/gsd-review`, `/gsd-fast`) are classified `hard-outlier` in the manifest and bypassed by the advisory runner
- [x] **RNNR-08**: Every deterministic workflow step sequence emitted by the advisory runner is stable for a fixed config snapshot; model output affects runtime reports, not advisory packet selection
- [x] **RNNR-09**: Dynamic-branch workflows are explicitly classified and tested through branch-specific harnesses rather than hidden model-dependent goldens
- [x] **RNNR-10**: Provider fallback is represented by typed runner transitions that record reduced confidence and missing providers atomically

### Query Registry FSM Integration

- [x] **QREG-01**: `gsd-sdk query fsm.state <workstream>` returns the current FSM state for the specified workstream
- [x] **QREG-02**: `gsd-sdk query fsm.run-id <workstream>` returns the active `runId` for the specified workstream
- [x] **QREG-03**: `gsd-sdk query fsm.transition <workstream> <toState> <outcome>` records the transition and emits an `FSMTransition` mutation event
- [x] **QREG-04**: `gsd-sdk query fsm.history <workstream>` returns transition history in chronological order
- [x] **QREG-05**: `gsd-sdk query thread.*` returns thread metadata without spawning a subprocess or parsing shell output
- [x] **QREG-06**: `gsd-sdk query phase.edit <workstream> <field> <value>` modifies phase state and emits the appropriate mutation event
- [x] **QREG-07**: Dotted-alias queries and spaced-alias queries dispatch to the same handler, verified by tests that assert identical output

### Nyquist P4 State

- [x] **P4NY-01**: The phase-chain FSM includes a `p4-compliance` state that executes after `verify` and before `advance`
- [x] **P4NY-02**: Entering `p4-compliance` emits a packet that routes to a single deterministic compliance pass, not triple-spawn verification
- [x] **P4NY-03**: The `p4-compliance` state is skipped when no Nyquist compliance check is configured for the workflow
- [x] **P4NY-04**: A parity test verifies that `p4-compliance` is never included in the `verify` multi-model packet sequence

### Error Handling and Observability

- [x] **ERRT-01**: Every failed FSM transition emits a typed error event with `code`, `message`, and `recoveryHint`
- [x] **ERRT-02**: Packet schema validation failure surfaces a machine-readable error naming failing field, `stepId`, and `workflowId`
- [x] **ERRT-03**: A missing `.planning/` directory causes the runner to emit a typed `init-required` error event, not an unhandled exception
- [x] **ERRT-04**: Stale lock detection surfaces a typed `lock-stale` event with lock holder identity and lock age in seconds
- [x] **LOGG-01**: Every FSM state transition is appended to workstream transition history with ISO-8601 timestamp, `fromState`, `toState`, `runId`, and `outcome`
- [x] **LOGG-02**: Reduced-confidence operations append `reducedConfidence: true` and `missingProvider: <name>` to the same atomic transition history entry
- [x] **LOGG-03**: Every successful `gsd-sdk compile` run logs command, workflow, agent, hook, and outlier counts

### Provider Fallback

- [x] **PROV-01**: When a provider is unavailable at a verification or auto-discussion step, the runner completes the transition using remaining providers and records `reducedConfidence: true`
- [x] **PROV-02**: Missing provider legs are recorded in transition history within the same atomic write as the transition itself â€” no indefinite stalls or hidden retries
- [x] **PROV-03**: `gsd-sdk query fsm.confidence <workstream>` returns `full` or `reduced:<missing-provider>` for the current run
- [x] **PROV-04**: Reduced-confidence transitions use typed FSM states and metadata, not ad hoc prose injected into packet instructions

### Hook Install / Build Path

- [x] **HOOK-01**: Running the supported install/build path copies all hooks from `hooks/` or `hooks/dist/` to expected install locations with executable permissions and without network fetches
- [x] **HOOK-02**: `scripts/build-hooks.js` compiles hook source files into `hooks/dist/` without network access
- [x] **HOOK-03**: The Codex hook is installed at the exact path and filename expected by the Codex CLI
- [x] **HOOK-04**: The uninstall manifest covers every installed hook, including `.sh` hooks
- [x] **HOOK-05**: Stale hook cache detection refreshes from source rather than skipping already-present files
- [x] **HOOK-06**: Hook install fixes are complete before `gsd-post-update` retirement is declared

### gsd-post-update Absorption

- [x] **UPDT-01**: Each audited `gsd-post-update` behaviour has a disposition record in the project requirements or generated manifest, not only in the seed spec
- [x] **UPDT-02**: Behaviours mapped to SDK/runtime dispositions are implemented as query handlers, FSM transitions, lifecycle hooks, packet fields, compiler checks, or provider capability transitions
- [x] **UPDT-03**: Behaviours mapped to installer dispositions are absorbed into `bin/install.js`, `scripts/build-hooks.js`, or hook manifest validation before retirement
- [x] **UPDT-04**: Running `gsd-post-update` against a migrated project produces no observable side effects â€” all patches it would apply are already present through supported surfaces
- [x] **UPDT-05**: No parity test invokes `gsd-post-update`; all parity tests pass without it

### Parity Test Suite

- [x] **PRTY-01**: The parity suite includes round-trip tests for every workflow classified deterministic, branch-specific tests for every workflow classified `dynamic-branch`, and explicit posture tests for every `hard-outlier`
- [x] **PRTY-02**: Every golden is generated from live `gsd-sdk compile` output or `gsd-sdk query` responses â€” no golden is hand-authored from Markdown prose
- [x] **PRTY-03**: Provider-fallback, filesystem-fallback, lock-staleness, and concurrent `STATE.md` write tests all run deterministically without network access
- [x] **PRTY-04**: HITL workflows use explicit mocked input seams to test suspension/resume paths without pretending user interaction is deterministic
- [x] **PRTY-05**: `STATE.md` mirror lock-protection tests prove concurrent write attempts return `lock-conflict`, not silent overwrite
- [x] **PRTY-06**: Golden staleness detection fails CI when live compile manifests diverge from committed parity fixtures without an intentional refresh
- [x] **PRTY-07**: All parity tests run in CI without external hosts, including no fetches to `raw.githubusercontent.com`
- [x] **PRTY-08**: Parity gate failures cause CI to exit non-zero; no assertion may be weakened to make the migration easier

## v1.x Requirements

These harden the extension surface after v1 parity and `gsd-post-update` retirement criteria are green. They do not supply v1-critical parity behaviour.

### Extension API

- **EXT-01**: An extension can insert a custom step into a workflow packet sequence at a declared position (`before:<stepId>` or `after:<stepId>`)
- **EXT-02**: An extension can replace the `instruction` field of an existing step by `stepId` without altering any other packet fields
- **EXT-03**: An extension can add a gate that the runner evaluates before emitting a specific step's packet; a failing gate blocks the transition and emits `gate-failed`
- **EXT-04**: An extension can register pre-transition or post-transition lifecycle hooks
- **EXT-05**: An extension can register provider availability checks for custom providers
- **EXT-06**: Extensions declare ordering dependencies via `before` and `after` arrays; dependency cycles are rejected with a diagnostic
- **EXT-07**: Extension step and field references use schema-stable identifiers, not positional indices

### In-Flight Migration / Rollback Hardening

- **MIGR-01**: A run-state file with `stateSchemaVersion` older than the SDK's current schema triggers `migration-required`
- **MIGR-02**: A run-state file newer than the SDK's current schema triggers `resume-blocked`
- **MIGR-03**: Transition history is preserved across in-place schema migrations
- **MIGR-04**: Rollback restores FSM run-state to the last `checkpoint: true` entry while preserving later entries as audit trail
- **MIGR-05**: State files created in v1 are forward-compatible with this migration tooling because they already include `stateSchemaVersion`

## v2+ Requirements

Deferred until v1 parity suite passes at 100% gate coverage. These clean up the surface; they do not change behaviour.

### Compatibility Cleanup / Markdown Slimming

- **SLIM-01**: Markdown workflow files fully superseded by FSM states are moved to `docs/archive/` only when `gsd-sdk compile --check-slim-eligibility <workflow-id>` verifies typed transitions, packet sequencing, provider routing, and parity coverage
- **SLIM-02**: Retained Markdown launcher files contain only the advisory invocation; inline workflow prose is removed only after eligibility passes
- **SLIM-03**: CI blocks any cleanup PR until all parity suite gates exit zero

### Hard Outlier Posturing

- **OUTL-01**: Each hard outlier (`/gsd-graphify`, `/gsd-from-gsd2`, `/gsd-ultraplan-phase`, `/gsd-review`, `/gsd-fast`) has a documented posture record stating its classification and migration rationale
- **OUTL-02**: Hard outliers are registered in the classification manifest with type `hard-outlier` so `gsd-sdk compile` does not report them as unknown workflows

## Out of Scope

| Feature | Reason |
|---------|--------|
| FSM implementation before spec roundtable review passes | Implementation follows `/gsd-map-codebase` and `/gsd-new-project --auto`; no code before design is locked |
| Tool execution inside the SDK | The SDK is a pure advisor; runtimes own filesystem, secrets, and tool calls |
| Default SDK-internal Claude Agent SDK model sessions | They use API billing rather than Claude Code OAuth/subscription execution |
| Markdown workflows as executable goldens | Markdown is prose, not deterministic input; goldens must come from live inventories |
| Weakening or deleting parity gate tests | Parity is the only mechanical proof of correctness; past regressions trace to this |
| Coarse packets | Reproduces the Markdown-prose problem in JSON form; rejected by compiler budget checks |
| Per-agent routing defaults as a substitute for per-transition hooks | Same agent gets different effort at different steps today |
| Runtime fetches for hook installation | Hook sources exist in `hooks/`; runtime fetches break offline and create version skew |
| Reopening locked design decisions on corpus discovery | `/gsd-map-codebase` is for missed factual context, not strategy revision |
| Separate persistence layer outside `.planning/` | `.planning/` remains the authoritative durable state location |
